Trojan-Go 鼓励开发传输层插件，以扩展承载协议类型并增加对抗审计的变化空间。

当前与插件功能最相关的实现位于：

- `src/internal/transport/layer/transport`
- `src/internal/app/wiring`
- `src/internal/core/relay/redirector`

传输层插件的作用，是替代默认传输层中的 TLS 承载逻辑，负责加密、混淆和承载 Trojan 流量。

插件与 Trojan-Go 之间只通过 TCP Socket 通信，本身没有框架级耦合。因此，你可以使用任何语言和实现方式开发插件。通常建议参考 [SIP003](https://shadowsocks.org/en/spec/Plugin.html) 约定，这样插件也更容易与 Shadowsocks 生态兼容。

开启插件后，Trojan-Go 与插件之间只交换 TCP 流量。插件可以进一步将这些 TCP 数据封装为任何你需要的协议，例如 QUIC、HTTP，甚至伪装成其他业务流量。

## 设计原则

Trojan-Go 对插件的设计预期主要有三点：

1. 插件本身应负责传输内容的加密、混淆、完整性校验，以及必要的重放防护。
2. 插件最好伪装为一种真实且常见的服务流量，而不是自定义一眼可辨识的新协议。
3. 服务端插件在发现内容篡改或疑似主动探测时，**不应直接断开连接**，而应将连接交回 Trojan-Go，由 Trojan-Go 继续执行回落或转发逻辑。

第三点尤其重要，因为这能充分利用 Trojan-Go 已有的抗主动探测能力。

## 为什么要交回 Trojan-Go

一个典型场景如下：

1. 假设插件伪装的是某种常见服务流量，例如 MySQL。
2. 审计方发现流量模式异常，主动连接服务器进行探测。
3. 插件校验后发现该连接不是合法代理流量，于是把连接交还给 Trojan-Go。
4. Trojan-Go 再将该连接转发到真实的目标服务，使探测方看到的行为仍然像一个正常服务。

这样做的目标不是让插件本身处理所有异常，而是让插件与 Trojan-Go 的回落机制协同工作。

## 实践建议

- 如果插件能完整满足加密、伪装和回落三项能力，效果最好。
- 如果暂时做不到，也仍然可以先实现可用版本，因为不常见的小众协议本身有时就足以规避通用审计。
- 但只要你替换了默认 TLS，就意味着安全性会更多依赖插件本身，因此应优先保证正确性和可审计性。
